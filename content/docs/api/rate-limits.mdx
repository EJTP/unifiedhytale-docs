The UnifiedHytale API implements rate limiting to ensure fair usage and platform stability.

## Limits

| Tier | Requests/Minute | Requests/Day |
|------|-----------------|--------------|
| Free | 60 | 1,000 |
| Creator | 120 | 10,000 |
| Partner | 300 | 100,000 |

## Rate Limit Headers

Every API response includes rate limit information:

```
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 58
X-RateLimit-Reset: 1704067200
```

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests per minute |
| `X-RateLimit-Remaining` | Remaining requests this minute |
| `X-RateLimit-Reset` | Unix timestamp when the limit resets |

## Handling Rate Limits

When you exceed the rate limit, you'll receive:

```json
{
  "success": false,
  "error": {
    "code": "RATE_LIMITED",
    "message": "Too many requests. Please try again later.",
    "retryAfter": 30
  }
}
```

HTTP Status: `429 Too Many Requests`

### Best Practices

1. **Cache responses** - Store data locally when possible
2. **Use webhooks** - Subscribe to events instead of polling
3. **Implement backoff** - Exponential backoff on 429 errors
4. **Batch requests** - Combine multiple operations when possible

### Example: Exponential Backoff

```javascript
async function fetchWithRetry(url, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    const response = await fetch(url);

    if (response.status === 429) {
      const retryAfter = response.headers.get('Retry-After') || Math.pow(2, i);
      await sleep(retryAfter * 1000);
      continue;
    }

    return response;
  }
  throw new Error('Max retries exceeded');
}
```
